command line

现在我们将会 command line 来进行 git 命令

🎯 克隆你的仓库到本地​

如果你已经在github上有自己的repository了 但是桌面上没有相对应的文件夹 那我们就需要克隆你的repository到桌面上

假设我们有个叫simpleTest的repository在GitHub上 我们一般采用以下格式来建立

热知识

clone下来的文件夹会自动初始化git 因此我们不需要再执行git init

现在打开你的命令行执行以下命令

# 切换到桌面目录
cd ~/Desktop

# 克隆你的仓库
git clone https://github.com/linsen25/simpleTest.git

执行完后，你的桌面上会有一个 simpleTest 文件夹。

初始化 git​

在开始为我们的项目链接通道之前我们需要先初始化 git 否则就无法进行下面的步骤

初始化 git 非常简单我们只需要一条命令

git init

✅ 输出示例：

Initialized empty Git repository in /Users/yourname/build/.git/

将 GitHub 仓库设置为远程仓库​

因为我们每个项目是独立的因此我们需要告诉我们的这个项目,也就是你正在工作的文件夹,通向 ssh 通道

我们一般采用以下格式来建立通道

git remote add orgin git@github.com:your_usrname/your_target_repository.git

比如说拿我的 username: linsen25. repository: linsen25.github.io 举例。应该是以下的形式

git remote add origin git@github.com:linsen25/linsen25.github.io.git

在添加了之后我们可以通过以下命令查看是否成功的链接了

git remote -v

✅ 输出示例：

origin  git@github.com:linsen25/linsen25.github.io.git (fetch)
origin  git@github.com:linsen25/linsen25.github.io.git (push)

提交到暂存区​

我们可以git add .将更改的代码提交到缓存区 他在commit之前可以无限重复

git add .

提交留言​

这里是我们在提交之前要留一句话这样我们才可以提交

这样是为了我们可以清楚的识别出我们都更改了什么在多人合作的场景下非常的重要

git commit -m "Initial commit"

查看以及更改分支​

在提交之前我们先来详细的了解一下分支

在推送阶段我们可以理解为是把本地的分支提交到远程的分支,首先我们先来看如何查看我们现在已有的分支

📌 查看分支​

git status这个命令可以帮助我们查看当前状态 然而git branch这个命令的前提是你的 repository 已经有了提交的记录才可以有显示也就是commit成功可以显示 如果说你 commit 不成功说明你没有提交所有的文件到暂存区也就是git add .再执行一遍 📋 详细解释：

查看当前状态

git status

查看本地分支

git branch

✅ 输出示例： 因为 default 是 master

*master

查看远程分支

git branch -r

✅ 输出示例：

origin/main
origin/feature

查看本地和远程

git branch -a

✅ 输出示例：

* main
  feature
  remotes/origin/main
  remotes/origin/feature

📌 创建分支​

✅ 创建本地分支

git branch <分支名>

✅ 创建远程分支

当你在本地创建一个新分支后，使用 git push origin <分支名> 命令时，Git 会自动在远程仓库中创建对应的分支。因此不需要远程分支

📌 切换分支​

git checkout <分支名>

📌 重命名分支​

✅ 重命名当前分支

git branch -m <新分支名>

✅ 重命名特定分支

git branch -m <旧分支名> <新分支名>

📌 删除分支​

✅ 删除本地分支

git branch -d <分支名>

✅ 强制删除本地分支 因为我们还没有合并他到主分支 为了以防丢失 git 会阻止你但是如果你就是想删掉可以强制

git branch -D <分支名>

✅ 删除远程分支

git push origin --delete <分支名>

🚀 如何建立 Git 本地分支和远程分支的连接？​

📖 什么是本地分支和远程分支的连接？​

在 Git 中，本地分支 和 远程分支 是两个不同的概念：

  • 本地分支：存在于你电脑上的分支，只能被本地访问和操作。
  • 远程分支：存在于远程仓库（如 GitHub、GitLab）中的分支，可以被团队成员访问和同步。

当你在本地创建一个新的分支后，需要将它推送到远程仓库，并建立一个 跟踪关系。这个跟踪关系可以让 Git 知道本地分支和远程分支之间的对应关系，从而简化你的操作。

🤔 为什么要建立连接？​

建立本地分支与远程分支的连接有以下几个好处：

  1. 简化 Git 操作
    建立连接后，你可以直接执行 git pull 或 git push，而不需要每次都指定远程分支的名称。

  2. 同步本地和远程分支
    连接本地分支和远程分支后，本地的更改可以同步到远程仓库，团队成员也可以拉取最新的代码。

  3. 避免 VS Code 提示“发布 Branch”
    如果没有建立连接，VS Code 会提示 “发布 Branch”，让你手动推送分支。

✅ 方法 1：在第一次推送时自动建立连接（推荐）​

  1. 确保你已经在本地创建了一个新分支：
git checkout -b feature-update
  1. 使用以下命令将本地分支推送到远程仓库，并自动建立连接：
git push -u origin feature-update
  1. 验证连接是否成功
git branch -vv
  1. 输出应该类似：
* feature-update f816302 [origin/feature-update] Resolved merge conflicts
main           02c5047 [origin/main] Update index.html in main branch

✅ 方法 2：手动设置跟踪关系​

如果你已经推送过分支，但忘记建立连接，可以使用以下命令手动设置：

git branch --set-upstream-to=origin/<your new branch> <your new branch>

git branch --set-upstream-to=origin/feature-update feature-update

✅ 方法 3：通过 VS Code 图形界面建立连接​

  • 打开 VS Code。

  • 进入 Source Control 面板。

  • 如果看到 “发布 Branch” 按钮，点击它。

  • VS Code 会自动执行：

git push -u origin feature-update

Github Pull Request 合并代码​

  • 合并 是将 两个分支的提交历史 整合为一个分支。
  • 合并后，主分支将包含功能分支中的所有更改。
场景说明
将功能分支合并到主分支完成新功能开发后，合并代码到主分支
将修复分支合并到主分支修复了 Bug 后，将修复代码合并到主分支
将开发分支合并到测试分支开发完成后，合并代码到测试分支进行测试

创建Github pull request (最推荐)​

🎯 Pull Request 的核心优势

  1. 代码审查（Code Review）
    Pull Request 提供了一个平台，允许团队成员在代码合并之前审查代码质量、提出改进意见，避免低质量代码直接进入主分支。

  2. 版本控制更安全
    直接在主分支（如 main 或 master）上提交代码可能会导致不可逆的错误，而 Pull Request 允许你先将代码提交到一个临时分支，确保代码无误后再合并到主分支。

  3. 更清晰的项目管理
    每个 Pull Request 都有一个独立的讨论空间，团队可以在这里讨论需求、优化方案、解决冲突，从而让项目进展更清晰。

  4. 自动化测试和 CI/CD 集成
    在创建 Pull Request 时，可以自动触发测试，确保代码不会破坏现有功能。很多团队会配置 CI/CD（持续集成/持续部署）流程，让每个 PR 都能自动运行测试。

Command Line 合并代码​

📌 合并分支的两种常用方法

✅ 方法 1：使用 git merge

这是最常用的合并方法，适合在 本地操作 时使用。

🛠️ 合并步骤：

1️⃣ 切换到主分支

git checkout main

2️⃣ 将功能分支合并到主分支

git merge feature-login

📋 解释：

git checkout main 切换到主分支。 git merge feature-login 将 feature-login 分支的更改合并到 main 分支。

✅ 方法 2：使用 Pull Request 合并 适合 多人协作 和 远程仓库 时使用。

1️⃣ 在本地推送功能分支到远程仓库

git push origin feature-login

2️⃣ 在 GitHub 或 Gitee 创建 Pull Request

选择 main 作为目标分支。 选择 feature-login 作为要合并的分支。 提交 Pull Request 并合并。

📌 查看哪些分支已经合并？​

✅ 查看已合并的分支

git branch --merged

✅ 查看未合并的分支

git branch --no-merged

📌 删除已合并的功能分支​

如果功能分支已经合并到主分支，可以删除这个分支：

✅ 删除本地分支

git branch -d feature-login

✅ 强制删除本地分支

如果分支未合并，但你确认不需要了，可以强制删除：

git branch -D feature-login

✅ 删除远程分支

git push origin --delete feature-login

📌 合并冲突及解决方法​

如果两个分支的更改有冲突，Git 会提示你手动解决冲突。

🛠️ 解决冲突步骤：

1️⃣ 合并时出现冲突提示

git merge feature-login

🔴 输出示例：

CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

2️⃣ 打开冲突文件，手动解决冲突

冲突文件中会标记冲突的部分，例如：

<<<<<<< HEAD
<h1>This is main branch</h1>
=======
<h1>This is feature-login branch</h1>
>>>>>>> feature-login

删除冲突标记，保留正确的内容。

3️⃣ 解决冲突后，添加文件到暂存区并提交

git add index.html
git commit -m "Resolve merge conflict"

📋 push 推送到远程分支​

  • git push 是将本地的 提交记录（commits） 推送到 远程仓库 的过程。
  • 当你在本地提交代码后，远程仓库不会自动更新，你需要使用 git push 命令将本地更改同步到远程仓库。
  • 首先，确保你在需要推送的分支上。

📋 常用的 git push 命令​

命令作用
git push origin main推送本地的 main 分支到远程仓库
git push origin <分支名>推送指定的分支到远程仓库
git push -u origin <分支名>设置默认的远程分支，方便后续推送
git push origin --delete <分支名>删除远程分支
git push --force强制推送，覆盖远程仓库的代码

🧩 第一次推送时的特殊操作​

如果你是第一次将本地分支推送到远程仓库，需要使用 -u 参数，设置默认的远程分支。

git push -u origin main

✅ 解释：

-u 的作用是将本地分支与远程分支关联起来，以后只需要使用 git push 就可以推送了。

🧩 强制推送的场景​

使用 git push --force 可以强制将本地分支的更改推送到远程仓库，覆盖远程分支的代码。

警告 ⚠️

强制推送会覆盖远程仓库的历史记录！请谨慎使用。

git push origin main --force

✅ 适用场景：

当远程分支有冲突或更新，需要覆盖远程分支的代码时。

回滚​

在 Git 中，你可以通过 回滚到之前的提交记录 来撤销更改、恢复代码到之前的状态。
以下是常见的 Git 回滚 方法，包括 git reset、git revert 和 git checkout 的用法。

在实际开发中，你可能会遇到以下情况：

  • 提交了错误的代码，希望回到之前的版本。
  • 代码有 Bug，想恢复到一个稳定的版本。
  • 误删除了文件或更改了重要内容，需要找回之前的状态。

通过 Git 的 回滚操作，你可以轻松解决这些问题。

✅ 查看提交历史​

git log

✅ 输出示例：

commit 62d8be8b9daddd0930c708a1c6df52a99cdb7e0f (HEAD -> main)
Author: linsen25 <liulinsen2517@gmail.com>
Date:   Wed Jan 10 2025

    Update project files

commit a12e92f5e5b6479a5c2815b9c4f18e8e1bd66e0b
Author: linsen25 <liulinsen2517@gmail.com>
Date:   Tue Jan 9 2025

    Initial commit

这里的 hash 值在后续的回滚的常用方法中会出现

🧩Git 回滚的 3 种常用方法​

✅ 方法 1：git reset（彻底回退）​

📋 使用场景： 当你想彻底抛弃某些提交，清除后续的提交记录。

# 回到指定的提交，并清除后续的提交记录
git reset --hard <commit-hash>

✅ 示例：

这里的 hash 值是git log显示的每次提交的 hash 值

git reset --hard a12e92f5e5b6479a5c2815b9c4f18e8e1bd66e0b

📋 解释：

--hard 表示回退到指定提交，并清除后续的更改。 ⚠️ 注意:git reset --hard 会删除回退后的更改，无法恢复！

✅ 方法 2:git revert(推荐)​

📋 使用场景：

当你想撤销之前的更改，但需要保留提交历史。

# 撤销指定的提交，并创建一个新的提交记录
git revert <commit-hash>

✅ 示例：

git revert 62d8be8b9daddd0930c708a1c6df52a99cdb7e0f

🎯 小总结： ✅ git revert 就是 “这次提交的更改不算了，但我会记得这次撤销”。

✅ 它会 再提交一遍一个新的版本，用来表示你撤销了之前的更改。

✅ 方法 3:git checkout(临时查看历史版本)​

📋 使用场景：

当你想临时查看某个历史版本的代码。

# 切换到指定的提交版本
git checkout <commit-hash>

✅ 示例:

git checkout a12e92f5e5b6479a5c2815b9c4f18e8e1bd66e0b

📋 解释：

git checkout 不会改变当前分支的状态，只是临时查看历史版本。

⚠️ 注意： 使用 git checkout 后，你需要使用 git switch 或 git checkout main 返回当前分支。
