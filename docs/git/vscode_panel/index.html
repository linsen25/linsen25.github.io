Vscode 图形化界面

在我们学习了git的原理以及command line形式的代码展示,现在我们要学习如何用vscode自带的源代码面板来简化git操作

⚠️注意

但是依旧是需要git init才会有这个tab

首先我们现在直接进入到vscode中在左侧的tab中我们可以找到我图片中的这个图标 我们可以看到俩个醒目的按钮打开文件夹和克隆仓库 也就是说我们可以直接从Github上把代码拷贝下来或者用你自己的文件夹 在下面的展示中我们依旧会用在command Line 章节我们用到的simpleTest repository

现在我们选择克隆仓库并选中simpleTest我们可以看到 在图片的左侧是我们之前提交的记录 右边是我们自己的代码

绑定连接​

在之前的command line我们就提到过 绑定远程仓库和本地仓库可以用vscode的图形化界面来进行绑定 操作也很简单我们只需要点击左侧面板的发布Branch就可以完成绑定不需要代码的使用现在我们来看下前后对比

在下面的图片中有4条命令分别是

  • 提交Branch前

git branch -r以及git branch -vv

git branch -r 告诉我们现在远程分支的Branch有哪些以及head指向的是谁也就是主分支

git branch -vv 显示分支的 最后一次提交信息，并且 显示分支的跟踪关系 我们可看到main 是 orgin/main说明绑定了 feature-update没有origin说明没有绑定

  • 提交Branch后 git branch -r以及git branch -vv

现在我们可以看到都是已经绑定了 也就是origin/<branch>的格式 同时远端的头指向是不变的

提交到暂存区​

在我们对文件作出更改的时候 在左侧面板标记更改的部分就会显示你更改的文件

现在我已经在图片上标记了一个红色框一个蓝色框

红色框​

我们可以理解红色框的更改是面对所有更改文件的 现在我们从左到右分析他4个按钮的功能

查看更改​

左数第一个是查看更改可以看到你对那些文件作出了哪些更改 并且你可以依旧在之中更改

撤销所有更改​

也就意味这删除你之前的更改 但是即便是追踪过也不可以回复你的文件作出的更改 只能回到上一个版本

注意

这个操作是不可逆的请谨慎使用

修改的文件数量​

最后一个按钮就是告诉你修改了多少个文件 仅此而已

添加所有文件到暂存区​

这个就很好理解了 就是执行git add .的操作

蓝色框​

在讲解完红色框的部分我们更好理解蓝色框了 他有三个按钮

展示本文件​

点击从左数第一个 按钮就会打开那个文件 仅此而已

放弃单文件更改​

这里是撤销单文件同样是不可逆的

添加文件到暂存区​

同理这里执行的就是git add <filename>的操作

提交留言​

提交留言在图形化面板中也是更加的便捷

在提交了到暂存区了之后 我们可以找到面板的的左上角 的提交按钮

1️⃣ 提交​

提示

默认行为：VS Code 默认执行的操作，如果不选择其他选项，仅仅提交到本地。

  • 功能：仅将暂存区中的文件提交到本地仓库，不会推送到远程仓库。

对应的 Git 命令:

git commit -m "Your commit message"

2️⃣ 提交（修改）​

  • 功能：将暂存区中的更改作为附加提交，追加到上一次提交中，而不是创建新的提交记录。

对应的 Git 命令:

git commit --amend
适用场景

如果你在上一次提交中忘记了一些更改，可以用此选项补充到之前的提交中。

危险

修改提交记录会改变提交的哈希值，可能导致远程分支的同步问题，建议在 尚未推送到远程仓库时使用。

3️⃣ 提交和推送​

  • 功能：将更改提交到本地仓库后，立即推送到远程仓库。

对应的 Git 命令：

git commit -m "Your commit message"
git push origin <branch-name>
适用场景

当你希望一次性完成提交和推送操作时使用。

4️⃣ 提交和同步​

  • 功能：在提交本地更改后，执行一次 拉取远程更改（pull），然后推送到远程仓库。

对应的 Git 命令：

git commit -m "Your commit message"
git pull origin <branch-name>
git push origin <branch-name>
适用场景

当你与团队协作开发时，确保自己的更改不会覆盖远程仓库的最新更改。

5️⃣ 提交并创建拉取请求​

  • 功能 将更改提交到本地仓库并推送到远程后，立即在远程仓库（如 GitHub）中创建一个 Pull Request。

  • 适用场景 当你完成了一个新功能或修复了一个问题，需要团队审查你的代码并合并到主分支时使用。

  • 背后操作： 执行提交并推送后，会自动打开一个用于创建 Pull Request 的窗口
